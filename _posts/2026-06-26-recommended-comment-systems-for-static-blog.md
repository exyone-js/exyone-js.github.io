静态博客本身没有后端，但评论互动又不能少。本站主站用 Twikoo 承载日常评论，杂谈分享区用 Giscus 补充高级互动功能。这篇文章聊聊它们各自适合什么场景，以及还有哪些备选方案值得关注。

---

## 为什么需要两套系统

本站目前同时跑着 Twikoo 和 Giscus，这不是为了折腾，而是两种系统各自覆盖了不同的需求。

**Twikoo** 是本站的默认评论系统。选它的核心理由只有一个：**读者不需要注册任何账号，打开页面就能直接评论**。你只需要填一个昵称和邮箱（邮箱甚至不需要验证），就能留下你的想法。对于那些只是路过、想说两句话就走的访客来说，这个门槛几乎为零。

相比之下，**Giscus** 基于 GitHub Discussions 构建，读者必须登录 GitHub 才能发言。这对于技术博客的受众来说似乎不算什么问题，但现实是：一位读者也许能稳定访问你的博客，却打不开 GitHub 的认证页面；另一个人可能只是想夸一句"写得好"，却要为此注册一个他平时根本用不到的账号。这种摩擦会直接损失评论——很多人看到登录弹窗就关掉了标签页。

但 Twikoo 的功能相对简单。它提供的是经典的一级评论 + 回复模式，不支持投票、问答标记、线程分类这些高级互动形式。所以本站的[分享杂谈区](/share/)选用了 Giscus 来补充：当你想发起一个讨论、给某个观点投票、或者用问答格式深入交流时，Giscus 的 Discussions 原生能力会更合适。

---

## Twikoo：低门槛的匿名评论

Twikoo 是一个专为静态网站设计的开源评论系统，由国人开发，文档和界面都是中文，配置过程的友好度很高。

### 数据存到哪里

很多人知道 Twikoo 支持 MongoDB，却忽略了它在裸机部署和 Docker 部署场景下的默认选择：LokiJS。

LokiJS 的工作方式非常直接：评论数据以 JSON 文件的形式存在服务器上，服务启动时把整个数据集加载到内存中，查询和写入都在内存里完成，再定期把变更写回到磁盘。

这种模式在评论量不大的时候，有几个明显的优势。最显著的优势就是没有外部依赖，你不需要再维护一个数据库服务；其次是延迟极低，因为所有操作都在内存中完成，不需要经过网络连接和查询优化器；最后是备份极其简单，一个 JSON 文件就可以完整保存所有评论。

当然，它也不是没有代价。LokiJS 不适合非常大的数据集，也不适合多个前端实例同时写入同一个数据源。但在个人博客这种场景下，这些限制几乎不会成为问题。我的博客评论量长期维持在几百条的量级，单节点部署，读写频率很低，这种情况下，LokiJS 的表现反而比 MongoDB 更轻量、更直接。

MongoDB 则完全是另一种思路。它是一个独立的数据库服务，支持索引、事务和并发控制，适合数据量大、访问频繁、或者需要多实例共享数据的场景。如果你的博客未来会扩展到多个前端节点，或者评论量增长到数万条以上，MongoDB 会是更合适的选择。

对于大多数体量有限的个人博客而言，MongoDB 的优势很难体现，反倒会让服务器背上不必要的维护负担与运行开销。尤其是在使用远程 MongoDB 数据库时，LokiJS 的本地优势会被进一步放大——所有的读写操作都在本地内存中完成，彻底规避了网络往返带来的性能损耗，响应更加直接。

### 优点与局限

**优点**：匿名评论门槛低、UI 简洁、反垃圾评论能力够用（自带 Akismet 和腾讯云内容安全）、支持邮件通知、支持表情和 Markdown、有完善的管理后台可以审核和回复。

**局限**：功能相对单一，没有投票、问答、分类这些 Discussions 级别的互动能力；需要自行维护后端，虽然部署简单，但终归是多了一个需要关心可用性的服务。

## Giscus：GitHub 原生的深度互动

Giscus 把评论数据存储在 GitHub Discussions 里，前端通过一个轻量组件嵌入到你的博客中。它的好处和限制都来自于同一个前提：**你的读者是 GitHub 用户**。

### 适合什么场景

如果你的博客受众主要是开发者，Giscus 的体验会非常顺畅。他们本来就有 GitHub 账号，登录过程几乎无感知。更重要的是，Discussions 的原生功能会自动带入你的博客：读者可以给评论点赞或点踩、可以把某个回复标记为"答案"（问答模式）、可以在不同的 Discussion Category 下分类留言。

本站把 Giscus 放在[分享杂谈区](/share/)，就是因为这个页面更像一个开放论坛，需要 Giscus 的投票和分类能力来组织话题，而不是简单的一级回复。

### 优点与局限

**优点**：完全免费、无广告、数据永久保存在 GitHub 上（不会消失）、功能丰富（反应表情、投票、问答、分类）、自动适配深色模式、加载速度快。

**局限**：强制 GitHub 登录，非技术读者几乎无法参与；需要公开仓库来存储 Discussions，虽然数据安全，但隐私敏感的内容不适合；中国大陆地区 GitHub 的访问稳定性时好时坏，会直接影响评论区的可用性。

## 其他值得了解的评论系统

如果你正在选型，以下这些系统也值得看看：

**Waline** —— Twikoo 的强劲替代品，同样支持匿名评论，但后端选择更丰富（LeanCloud、腾讯云、Deta、自建等）。功能上比 Twikoo 更全，支持页面浏览量统计、多语言、丰富的通知渠道（邮件、Telegram、微信等）。如果你需要比 Twikoo 更强大的管理能力，Waline 是自然的下一步。

**Disqus** —— 最老牌的商业评论系统，接入最简单，几行代码就能跑起来。但免费版有广告，而且广告越来越重。另外 Disqus 在中国大陆基本不可用，如果你的读者主要在国内，可以直接排除。

**Gitalk / Utterances** —— 和 Giscus 类似，都是基于 GitHub 的免费方案。Gitalk 用 Issues 存评论，Utterances 也是 Issues，而 Giscus 用 Discussions。相比之下 Giscus 功能更现代，反应表情和分类是 Discussions 独有的，建议新项目直接选 Giscus。

**Cusdis** —— 一个主打轻量和隐私的评论系统，后端可以自己托管，前端非常精简。它支持匿名评论，管理后台简洁直观。缺点是社区和生态还在早期，功能更新比较慢。

**Remark42** —— 偏技术向的自托管方案，功能很全（匿名、社交登录、通知、管理员模式），但部署门槛明显比 Twikoo 高，适合有运维经验的用户。

## 选型建议

如果你和我的情况类似——博客面向混合受众，既有技术读者也有普通访客——**Twikoo + Giscus 的组合**是一个务实的选择。日常文章用 Twikoo 兜底，保证任何人都能参与；需要深度互动的特殊页面（比如问答、投票、杂谈）再用 Giscus 补充。

如果你的受众几乎全是开发者，且都能稳定访问 GitHub，**单用 Giscus** 就足够了，数据放在 GitHub 上省心省力。

如果你更看重功能丰富度，又不介意多折腾一点后端配置，**Waline** 是比 Twikoo 更全面的选择。

如果你完全不想维护任何后端，也不想绑定 GitHub，**Cusdis** 或 **Disqus**（能接受广告的话）是最省事的方案。

---

评论系统的选择没有绝对的最优解，关键是匹配你的读者群体和你愿意承担的维护成本。本站的两套系统跑到现在，各自发挥了应有的作用——Twikoo 接收了绝大多数日常留言，Giscus 则在分享区支撑了几次还算热烈的讨论。如果你也在为静态博客选评论系统，希望这篇文章能帮你少走一点弯路。
